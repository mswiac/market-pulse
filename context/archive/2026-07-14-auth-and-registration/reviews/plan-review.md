## Findings

### F1 — 600,000 PBKDF2 iterations exceeds workerd's hard cap of 100,000

- **Severity**: ❌ CRITICAL
- **Impact**: 🔬 HIGH — architectural stakes; think carefully before deciding
- **Dimension**: End-State Alignment
- **Location**: Phase 3, item 1 (`src/worker/lib/password.ts`)
- **Detail**: Cloudflare's `workerd` runtime hard-caps `crypto.subtle` PBKDF2 at 100,000 iterations (unconfigurable, confirmed via `workerd` source and `cloudflare/workerd#1346`, still open in 2026). At 600,000 iterations every register/login call throws `DOMNotSupportedError`, on any billing plan — this also breaks Phase 3's own automated test success criterion. Follow-up discussion with the user further established that even 100,000 iterations costs ~50ms CPU (Cloudflare's own estimate), well over the Workers Free plan's ~10ms budget this project is currently on (confirmed in `context/foundation/infrastructure.md`'s risk register) — and the user confirmed the project is staying on the Free plan.
- **Fix A ⭐ Applied**: Tune iterations down to a value that fits the Free plan's CPU budget (default 10,000, within NIST SP 800-63B's floor recommendation; benchmark and raise toward the 100,000 ceiling in Phase 3 if headroom allows).
- **Fix B (combined with A)**: Add an HMAC-SHA256 pepper keyed by a new `PASSWORD_PEPPER` Workers Secret, to compensate for the lower iteration count at negligible extra CPU cost — a leaked D1 dump alone is insufficient to brute-force passwords without also compromising the Workers Secret.
- **Decision**: FIXED (Fix A + Fix B combined, per user's explicit choice after discussion). Plan updated: Key Discoveries, Critical Implementation Details, Phase 3 item 1 contract, Phase 3 item 4 contract (Env gains `PASSWORD_PEPPER`), Phase 3 manual verification (+3.6 secret setup, +3.7 CPU measurement), Performance Considerations, Testing Strategy, plan-brief.md Key Decisions + Open Risks + Phases-at-a-Glance.

### F2 — `npm run typecheck` will fail on the new test file (missing ambient types)

- **Impact**: 🏃 LOW — quick decision; fix is obvious and narrowly scoped
- **Dimension**: Plan Completeness

### F4 — No handling for a session expiring while the user is already on the page

- **Severity**: ⚠️ WARNING
- **Impact**: 🔎 MEDIUM — real tradeoff; pause to reason through it
- **Dimension**: Blind Spots
- **Location**: Phase 4, items 3–4 (`AuthService`, `auth.guard.ts`)
- **Detail**: `AuthService.isAuthenticated` is only set at bootstrap (`checkSession()`) and re-checked at navigation time by the guard. A session expiring/revoked while the user sits on the guarded home view goes undetected until the next navigation or refresh; S-02's future API calls would also have nowhere to route a 401.
- **Fix**: Add a functional `HttpInterceptor` (`sessionExpiredInterceptor`) that catches any `401`, clears `AuthService.currentUser`, and navigates to `/login`; register via `withInterceptors()` alongside `provideHttpClient`. Reusable by S-02.
- **Decision**: FIXED. Added as Phase 4 item 9, plus manual verification step (session-row deletion test) and Progress item 4.10.
